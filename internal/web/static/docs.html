1What is goon

goon is an autonomous AI engineer in a single Go binary. It can act as a one-shot CLI agent (you ask, it executes) or run as a background daemon that polls a ticket board (Jira / GitHub Issues), plans each ticket, asks you to approve, codes it, tests, verifies, updates its own memory, and opens a PR.

You can drive it from three places, all backed by the same memory store:

  • CLI — goon "task", goon start, goon train, etc.
  • Web UI — this page; the dashboard at / mirrors every CLI command.
  • Telegram bot — DM your bot to monitor, approve, review PRs, or chat with the model.

Why goon is different

  • Single binary, zero deps. Go stdlib only. Drop the binary anywhere, run.
  • Pluggable everything. LLM (OpenAI, Anthropic, Ollama, mock), board (Jira, GitHub), git host (GitHub, GitLab, Bitbucket).
  • Safe by default. Strict-JSON tool contract, regex blocklist on every shell command, approval gates before code lands.
  • Persistent memory. Markdown notes the agent reads and writes across runs.

2Quick start

Pick the path that matches how you want to use goon. All paths share the same setup steps; only the env vars differ.

Path A — Offline test

No keys, no network. See the agent loop work.

GOON_LLM_PROVIDER=mock \
GOON_MOCK_REPLIES='{"tool":"finish","args":{"message":"hello"}}' \
goon "say hi"

Path B — One-shot CLI agent

Use goon as a personal assistant for ad-hoc tasks.

goon config set GOON_LLM_PROVIDER openai
goon config set OPENAI_API_KEY sk-...
goon "tidy go.mod" --explain

Path C — Autonomous daemon

Polls a board, asks approval, files PRs.

goon config set GOON_BOARD jira
goon config set GOON_GIT_HOST github
goon config set ATLASSIAN_BASE_URL https://acme.atlassian.net
goon config set ATLASSIAN_EMAIL me@acme.com
goon config set ATLASSIAN_API_TOKEN ...
goon config set GITHUB_TOKEN ghp_...
goon start --web=:8080

Path D — Telegram bot

Drive goon from your phone.

goon config set TELEGRAM_BOT_TOKEN ...
goon config set GOON_TELEGRAM_SECRET 
goon start
# DM /auth <random-phrase> to your bot

Verify your setup

goon doctor live-probes every configured provider. Anything red is missing creds — fix in .env and re-run.

goon doctor

3Configuration

All knobs are environment variables (or .env). The config file lives at ~/.config/goon/.env. Manage it with the config subcommand: 

goon config              # show all (secrets masked)
goon config set KEY VAL  # KEY=VAL also works
goon config get KEY
goon config edit         # open in $EDITOR
goon config path         # print the config file path

Values exported in your shell always win over the file. goon config show labels each row [shell], [config-file], [default], or [unset].

Storage layout

Per-project state under ./storage/. Each working directory gets self-contained state you can .gitignore.

./storage/
├── memory.json        passive runtime state
├── memory/            active markdown notes (PINNED.md, …)
├── logs/goon.log      rotated structured log
└── goon.pid           present while daemon runs

Override the root with GOON_STORAGE_DIR to share state across repos.

5The autonomous workflow

When goon start picks up a ticket from the board, it runs this resumable pipeline:

1. Triage          LLM produces an ordered plan + suggested repo
2. Confirm repo  ◀ gate: "Confirm repo for ENG-123? (yes / change=<path> / no)"
3. Approve plan  ◀ gate: "Approve work + test plan?  (yes / no)"
4. Execute         agent runs each plan step, safety-validated
5. Test            best-effort `make test` (or repo-defined test command)
6. Verify × N      LLM re-checks the work N times (default 3)
7. Update memory   agent distils learnings into PINNED.md / topic notes
8. Open PR         GitHub / GitLab / Bitbucket
9. Notify          Telegram + board comment + status transition

The pipeline is a resumable state machine. When a gate fires, the workflow queues a question and exits; the daemon picks the workflow up again on a later tick once you reply.

Set GOON_AUTO_APPROVE=1 or auto_approve: true in workflow.json to skip the gates entirely for fully unattended runs.

6Approval gates

Two gates fire by default before any code lands:

confirm_repo

After triage, goon proposes a target repo and asks you to confirm. Reply with yes to accept, or change=/path/to/repo to override.

approve_plan

After repo confirmation, goon shows the numbered plan it produced and asks you to approve. Reply yes to proceed, no to abort.

How to answer

Three equivalent ways:

  • This web UI — pending questions appear on the dashboard. Click Answer.
  • CLI — goon train walks you through pending questions, or goon train answer <id> <text> for non-interactive use.
  • Telegram — /queue to see what's pending, /answer <id> yes to approve.
Anything other than yes at approve_plan aborts the workflow. The ticket is marked failed in memory.json; the board comment + transition do not happen.

7Memory

goon has two memory layers that serve different jobs.

Passive memory (memory.json)

Runtime state. Don't edit by hand — goon manages it.

Holds:

  • Last 200 interactions and a per-command frequency counter
  • Tickets the daemon has seen
  • Workflow records (with state, stage, plan, approvals, PR URL)
  • The question queue (asked, answered, pending)
  • Daemon status snapshot
  • Authorized Telegram chat IDs

Active memory (markdown notes)

Plain .md files at ./storage/memory/. The agent reads + writes them via tool calls. Knowledge survives across runs and any LLM provider can use them — they're just markdown.

goon memory init               # creates the dir + seeds PINNED.md
goon memory list               # * = auto-loaded
goon memory edit PINNED.md
goon memory read learnings/regex.md
goon memory search "auth"      # case-insensitive grep

Five tools the agent gets automatically

ToolPurpose
memory_listenumerate every note
memory_readfetch one note's contents
memory_writecreate or REPLACE a note
memory_appendadd to an existing note (preferred)
memory_searchcase-insensitive substring scan

PINNED.md — the always-loaded note

When ./storage/memory/PINNED.md exists, its contents are injected into the agent's system prompt on every call. Use it for facts that should always be top-of-mind: codebase conventions, names of services/people, "don't do this" rules. Keep it short and high-signal.

9Customize the pipeline

Drop a workflow.json in your repo root and goon picks it up on the next poll — no rebuild, no restart.

goon workflow init     # writes a starter ./workflow.json
goon workflow show     # prints the resolved config
goon workflow edit     # open in $EDITOR
goon workflow hooks    # list every hook name + env vars

Minimal example

{
  "version": 1,
  "name": "engineering-prod",
  "branch_prefix": "feature/",
  "test_command": "make ci",
  "verify_runs": 5,
  "auto_approve": false,
  "pr_title_template": "FIX({{.Key}}): {{.Title}}",
  "pr_body_template": "Resolves {{.Key}}\n\nBranch: {{.Branch}}",
  "extra_labels": ["customer-x"],
  "hooks": {
    "before_execute": ["echo 'goon picked up {{.Key}}'"],
    "before_test":    ["make build"],
    "before_pr":      ["go fmt ./...", "goimports -w ."],
    "on_failure":     ["echo \"goon failed on $TICKET_KEY\" | mail -s goon you@x"]
  }
}

Hook commands run sequentially through sh -c in the repo dir, with ticket env vars exported ($TICKET_KEY, $TICKET_URL, $REPO, $BRANCH, etc.) and Go-template substitution inside the command itself. Failed hooks fail the workflow phase. on_failure is best-effort.

Replace the pipeline (declarative stages)

For non-engineering work — marketing, sales, ops — you can replace the built-in pipeline wholesale with a stages array:

{
  "version": 2,
  "stages": [
    {"name": "triage",  "type": "llm",   "json_mode": true,
     "prompt": "Break {{.Key}} into 3-7 atomic steps. Reply JSON {\"steps\":[{\"title\":\"...\"}]}."},
    {"name": "execute", "type": "agent",
     "task": "Implement: {{(index (index .Stages.triage.steps 0) \"title\")}}"},
    {"name": "verify",  "type": "agent", "repeat": 3,
     "task": "Verify {{.Key}} is done. List defects via finish."}
  ]
}

Stage types are llm or agent. Each stage's output is available to later stages via {{.Stages.<name>.field}}.

10Tools shipped

The agent picks tools by name from this manifest each turn:

read_fileup to 64KB

memory_*five tools for the markdown notes store
finishend the loop with a summary

The model contract

The LLM must emit exactly one JSON object per turn:

{
  "tool": "run_command",
  "args": { "command": "ls -la internal" },
  "rationale": "list packages before reading"
}

Parse errors are fed back to the model so it self-corrects. Unknown tool names are rejected.

12Safety model

Three layers — each independent, all must pass.

1. The strict-JSON contract

Every model output must parse as a ToolCall. Bad JSON is fed back as a parse error so the model self-corrects.

2. The regex blocklist

internal/safety blocks the most dangerous patterns regardless of mode: rm -rf /, rm -rf ~, mkfs.*, dd of=/dev/*, fork bombs, shutdown/reboot/halt, chmod -R 777 /, curl … | sh. Hooks go through the same validator.

3. Mode-aware execution

In --run mode the executor asks y/N before any mutating tool call. Default mode is dry-run — prints what would happen, never executes.

4. Approval gates

In daemon mode, no code lands without confirm_repo + approve_plan approval (unless you set GOON_AUTO_APPROVE=1).

13Troubleshooting

"goon doctor" all green but the daemon never opens a PR

Check that GOON_GIT_HOST is set. doctor reports "skipped" (not red) when it's missing. Set the var, supply matching auth (e.g. GITHUB_TOKEN), restart.

"goon start" prints "waiting for config" forever

The daemon needs GOON_LLM_PROVIDER AND GOON_BOARD at minimum. Run goon doctor; anything red is missing creds.

The daemon stops at "awaiting approval" and never resumes

That's a gate firing. Answer the question via this web UI's queue panel, or goon train, or your Telegram bot's /answer. The daemon resumes on the next poll cycle (default 5 min — set GOON_POLL_SECONDS=30 for faster feedback during testing).

Telegram bot says "PR review unavailable" or "not yet implemented"

The first means GOON_GIT_HOST isn't set. The second means the configured host's adapter doesn't support review yet — currently only github and bitbucket do; gitlab is read-only.

Telegram bot accepts /auth but no commands appear in the menu

Telegram caches the menu briefly per client. Close and reopen the chat. The bot logs → telegram bot: 16 commands registered in menu at startup if registration succeeded.

"goon train" shows no questions but the workflow paused

goon train --all shows answered + pending; goon train only shows pending. If both are empty, the workflow may have failed before queueing — check goon logs --tail=50.

Want faster feedback during testing 
GOON_POLL_SECONDS=15 GOON_VERIFY_RUNS=1 GOON_AUTO_APPROVE=1 goon start

Tight poll interval, single verify pass, no approval gates. Don't ship like this.

How to wipe everything and start fresh 
goon stop
rm -rf ./storage          # clears state for THIS project only
# OR for global clean:
rm -rf ~/.config/goon
goon uninstall --yes --purge   # also removes the binary

14FAQ

Does goon work on Windows?

Yes — both CLI one-shot mode and daemon mode are supported on Windows alongside Linux and macOS. Workflow hooks and run_command use cmd /C on Windows and sh -c on POSIX, so cross-platform hook scripts should either stick to commands both shells understand or branch on the platform inside the hook. The same multi-process memory.json caveat applies on every OS: a sibling lockfile coordinates writes, and on filesystems that don't support atomic file creation reliably (some network mounts) you may see a "lockfile held by another process" warning. WSL also works fine if you prefer a POSIX environment.

Can I run goon without an LLM API key?

Yes — use Ollama for local models, or the mock provider for offline testing. Both are zero-cost.

How does goon decide which repo to work on?

By GOON_REPO_MAP (mapping board project keys to local paths) or by the ticket's project field. The confirm_repo gate gives you a chance to override before any code is written.

What does goon write to memory.json that I should know about?

Tickets seen, workflow records (state + plan + approvals + PR URL), the question queue, daemon status snapshot, and authorized Telegram chat IDs. No source code, no secrets — secrets stay in .env, never in memory.json.

Can I share state across projects?

Yes — set GOON_STORAGE_DIR=~/.goon-state to a shared path. Default is per-project (./storage/) so each repo gets clean state.

What if the LLM hallucinates a tool call?

It's rejected (the agent loop validates against the registered tool list and feeds the error back). After max steps without a valid call, the agent surrenders with a "max steps reached" message rather than executing anything random.

How do I add a new tool?

Implement tools.Tool and register in DefaultRegistry. The model picks it up automatically through the manifest line in the system prompt.

How do I add a new LLM provider?

Implement llm.Provider and route in llm.NewFromEnv. The agent loop is provider-agnostic.

How do I add a new git host (e.g. Gitea)?

Implement githost.Host and route in githost.NewFromEnv. To enable PR review via the Telegram bot, also implement the githost.PRReviewer companion interface.
